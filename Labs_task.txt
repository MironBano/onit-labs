Лабораторная работа №2:
Docker – это инструмент для контейнеризации, предназначенная для автоматизации разработки, доставки и запуска приложений в изолированных средах – контейнерах.
Контейнер – это изолированное пространство, которое позволяет запускать приложения с их зависимостями отдельно от основной системы.
Docker обеспечивает изоляцию приложений, создавая отдельные пространства для их работы. Каждый контейнер использует образы, которые запускаются в изолированном окружении, но при этом работают на общем ядре операционной системы. Это достигается за счет использования технологии виртуализации на уровне ядра.
Docker-образ – это неизменяемый шаблон, содержащий код, библиотеки, зависимости и настройки, необходимые для запуска приложения в изолированном контейнере.
Пример Doker-образа ниже:
FROM python:3.9 
WORKDIR /app 
COPY requirements.txt . 
RUN pip install --no-cache-dir -r requirements.txt 
COPY . . 
CMD ["python", "app.py"]

В этом примере используется официальный образ Python версии 3.9 через команду FROM. Команда WORKDIR определяет рабочую директорию, где расположен проект с кодом. Команда COPY requirements.txt добавляет файл зависимостей в контейнер, а затем выполняется их установка с помощью pip install (команда RUN). После этого проект копируется в контейнер, и запускается основной скрипт приложения app.py через команду CMD.
Пример основных команд:
docker-compose build : собрать или пересобрать образы для сервисов.
docker-compose up : запустить контейнер.
docker-compose down: остановить контейнер.
docker-compose ps : посмотреть статус контейнеров.
docker-compose logs : посмотреть логи всех контейнеров.
docker-compose logs -f : отслеживать логи в реальном времени.

Стейдж сборка (Multi-stage builds) – это возможность использовать несколько инструкций FROM в одном Dockerfile. Основная суть заключается в «чистой» сборке образа приложения: сначала собирается изначальный образ со всеми зависимостями (библиотеками, компиляторами и т.д.), а затем результат этой сборки (например, готовый bin файл или окружение) копируется в финальный (чистый) образ.
Преимущества такого подхода:
1.	Уменьшение размера образа;
2.	Безопасность;
3.	Скорость деплоя;
Пример стейдж сборки (в Dockerfile):
# --- Сборка зависимостей - это условно "грязный" образ ---
FROM python:3.9 AS builder
WORKDIR /app
# Устанавливаем инструменты для компиляции (если нужны для pip пакетов)
RUN apt-get update && apt-get install -y gcc python3-dev
COPY requirements.txt .
# Устанавливаем зависимости в отдельную папку
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

# --- "Чистый образ" ---
# Используем slim версию (она легче обычной)
FROM python:3.9-slim
WORKDIR /app
# Копируем установленные пакеты из этапа builder
COPY --from=builder /install /usr/local
# Копируем исходный код
COPY . .
CMD ["python", "app.py"]

Для управления несколькими контейнерами используется файл docker-compose.yml.
Пример docker-compose.yml. :
services:
  app:
    # Указываем, что образ нужно собрать из Dockerfile в текущей папке
    build: .
    
    # Имя контейнера (опционально, но удобно для управления)
    container_name: my_python_app
    
    # Проброс портов (хост:контейнер). 
    ports:
      - "8000:8000"
    
    # Политика перезапуска. Контейнер будет перезапускаться, если упадет
    restart: unless-stopped
    
    # Переменные окружения
    environment:
      - PYTHONUNBUFFERED=1  # Чтобы логи Python сразу попадали в stdout docker
      - ENV_VAR_NAME=value  # Пример переменной

depends_on – это директива в docker-compose.yaml, которая управляет порядком запуска сервисов. Эта директива выставляет зависимость одного сервиса от другого, т. е. например, пока сервис с БД не будет запущен, не будет запущен и сервис с основным приложением.
Пример реализации директивы:
services:
web:
build: .
ports:
- "8000:8000"
# web запустится ПОСЛЕ того, как запустится контейнер db
    		depends_on:
      	- db

db:
image: postgres:13
environment:
POSTGRES_PASSWORD: example

И казалось бы, все замечательно в таком подходе. Однако есть один важный нюанс – depends_on гарантирует только запуск контейнера с сервисом, но не готовность сервиса. Может произойти такая ситуация, что Docker запустит контейнер с БД; он будет запушен (статус UP), однако сама БД может еще инициализироваться некоторое время (несколько секунд). Но Docker уже запустит контейнер с приложением, которое будет сразу пробовать подключаться к БД, которая еще не доступна. Так приложение может упасть с ошибкой.
Чтобы избежать такой потенциальной ситуации можно настроить health-check.
Пример реализации health-check:
services:
  db:
    image: postgres:13
    environment:
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: ${POSTGRES_DB}
    volumes:
      - pgdata:/var/lib/postgresql/data
    # 1. Настраиваем проверку здоровья для БД
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 5s
      retries: 5

  web:
    build: .
    ports:
      - "8000:8000"
    # 2. Указываем, что ждем именно здорового статуса
    depends_on:
      db:
        condition: service_healthy
    environment:
      DATABASE_URL: postgresql://postgres:example@db:5432/mydb

CI/CD пайплайны (конвейер) – это автоматизированный процесс доставки проекта от разработчика к пользователю, включающий сборку, тестирование и разворачивание (деплой) приложения.
Такой конвейер состоит из двух основных этапов: Continuous Integration (CI) и Continuous Delivery/Deployment (CD).
Более подробно про эти этапы и их шаги можно посмотреть ниже:
 

Continuous Integration (CI) – задача данного этапа в постоянной выполнения тестов для выявления проблемных мест и багов кода, сборке и создания образов проекта.
Continuous Delivery/Deployment (CD) – на данном этапе проект разворачивается из собранного образа (артефакта) на инфраструктуре (тестовой или продуктовой).
Отличие Continuous Delivery от Continuous Deployment заключается в том, что в первом случае деплой происходит ручным способом, а во втором – автоматически.

Для создания такого конвейера потребуется отдельный .yml файл, который должен лежать в каталоге проекта по такому пути:
•	для GitHub (.github/workflows/ci-cd.yml)
•	для GitLab (.gitlab-ci.yml)

Ниже расположен пример ci-cd.yml файла для GitHub:
name: CI/CD

on:
  push:
    branches: [ master ]
  pull_request:
    branches: [ master ]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  test:
    name: Run Tests
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
        
      - name: Setup Python + uv  
        uses: astral-sh/setup-uv@v3
        with:
          python-version: "3.12"
          enable-cache: true
        
      - name: Install dependencies
        run: uv sync --dev
        
      - name: Run tests
        run: uv run python -m pytest -v --tb=short
        env:
          postgres_user: test_user
          postgres_password: test_password
          postgres_db: test_db
          postgres_host: localhost

  build:
    runs-on: ubuntu-latest
    needs: test
    if: github.event_name == 'push'
    
    permissions:
      contents: read
      packages: write
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3
      
      - name: Log in to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      
      - name: Extract Docker metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=ref,event=branch
            type=ref,event=pr
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            type=sha,format=short
            type=raw,value=latest,enable={{is_default_branch}}
      
      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

Блок с директивами on.push описывает при каких действиях пайплайн запускается – в данном случае при пуше и пулреквесте в master ветку.
Блок с директивами env – глобальные переменные окружения. В данном случае хранят адрес GitHub Container Registry и имя Docker-образа (берётся из репозитория).
Далее в файле расположены директива с jobs – это этапы, которые проходят во всем пайплайне:
test – запускает тесты (CI этап в данном примере):
•	клонирует проект в раннер (виртуальную машину, которая выполняет все задачи в GitHub Actions);
•	устанавливает среду и зависимости;
•	запускает сами тесты;
build – запускает сборку и публикацию образа (CD этап в данном примере):
•	определяет права доступа;
•	клонирует проект в раннер;
•	конфигурирует Docker;
•	авторизируется в GHCR (GitHub Container Registry);
•	генерирует метаданные и теги;
•	cобирает и публикет образ.

Секреты и переменные
Так как код проекта обычно хранится в репозиториях, то часть определенных переменных и секретов также необходимо хранить в них. Для этого на GitHub есть GitHub Actions (Репозиторий → Settings → Secrets and variables → Actions), а для GitLab если GitLab CI/CD (Проект → Settings → CI/CD → Variables → Expand).

В Secrets размещаются учетные данные пользователя БД, API-ключи, приватные-ключи, токены.
В Variables – URL, флаги, имена окружений и другие нечувствительные данные.

Тесты при сборке
Тесты в CI позволяют автоматизированно проверять код при каждом изменении (пуш, пул-реквест), гарантируя, что новая версия не сломает существующие функциональности проекта.
Типы тестов:
•	Unit-тесты (тестирование отдельной функции или класса);
•	Integration-тесты (тестирование на взаимодействие каких-то компонентов);
•	E2E-тесты (тестирование полного пользовательского сценария);
•	Lint/Type checks (проверка стиля кода (через линтеры)).

Nginx – веб-сервер с открытым исходным кодом, который используется для быстрой обработки большого количества одновременных запросов, обслуживания статического контента (картинки, CSS, JS), распределения нагрузки (балансировки) между серверами и защиты сайтов (SSL терминация).

 

Основной файл конфигурации Nginx – nginx.conf. Конфигурация состоит из контекстов (блоков), вложенных друг в друга:
•	main – глобальные настройки (пользователь, количество рабочих процессов).
•	events – настройки обработки соединений.
•	http – настройки HTTP-трафика. Внутри него располагаются:
o	upstream – группа бэкенд-серверов для балансировки.
o	server – описание виртуального хоста (домена).
o	location – правила обработки конкретных URI внутри сервера.
Пример простой конфигурации nginx.conf:
http {
    upstream backend_pool {
        server app1:80;
        server app2:80; 
        server app3:80; 
    }

    server {
        listen 80;
        
        location / {
            proxy_pass http://backend_pool;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }
    }
}

Директива listen 80; – означает, что сервер принимает HTTP-запросы на 80 порту всех интерфейсов.
Директива proxy_pass – перенаправляет запрос на группу серверов backend_pool
Директива proxy_set_header Host $host; – передаёт оригинальное имя хоста из запроса клиента.
Директива proxy_set_header X-Real-IP $remote_addr; – передаёт реальный IP-адрес клиента. Без этого бэкенд увидит только IP балансировщика.

Методы балансировки нагрузки
Балансировка настраивается в блоке upstream. Nginx поддерживает следующие алгоритмы:
1.	Round Robin (по умолчанию): Запросы распределяются по серверам последовательно. Если сервер падает, он исключается из ротации. Пример настройки
upstream application {
   server 172.25.208.1:5002;
   server 172.25.208.1:5001;
}
Чтобы избежать простоя серверов, можно использовать некоторый числовой приоритет. У каждого сервера появляется свой вес, который определяет, как много трафика распределяется на конкретный экземпляр приложения (чем больше число, тем большая доля запросов будет направляться на этот сервер). Таким образом мы гарантируем, что более мощные серверы получат больше трафика.

upstream application{
      server 10.2.2.11 weight=5;
      server 10.2.2.12 weight=3;
      server 10.2.2.13 weight=1;
   }
2.	Least Connections (least_conn) – обеспечивает равномерное распределение нагрузки между экземплярами приложения, как раз основываясь на количестве соединений с сервером. Такой подход эффективен для долгих запросов. Чтобы его настроить, в блоке upstream надо указать least_conn.
upstream application{
      least_conn;
      server 10.2.2.11;
      …
  }
3.	IP Hash (ip_hash). Этот метод работает на основе IP-адреса клиента. Он гарантирует, что все запросы с одного адреса будут доставлены на один и тот же экземпляр приложения. Алгоритм работает следующим образом: высчитывает хэш у адреса клиента и адреса сервера, и использует этот результат как уникальный ключ при балансировке.  Необходимо для поддержки сессий (sticky sessions).
upstream app{
      ip_hash;
      server 10.2.2.11;
      …
   }

При настройке балансировщика также важно обнаруживать неполадки с серверами, и, в случае чего, прекращать направлять трафик на «упавшие» экземпляры приложения. 
Чтобы балансировщик мог пометить адрес сервера как недоступный, необходимо определить дополнительные параметры в блоке upstream: failed_timeout и max_fails.
•	failed_timeout – тут указывается время, в течение которого должно произойти определённое количество ошибок соединения, чтобы адрес из блока upstream стал помечен как недоступный.
•	max_fails – задаётся это самое количество ошибок соединения.
Пример настройки:
upstream application{
      server 10.2.0.11 max_fails=2 fail_timeout=30s;
      …
 }
