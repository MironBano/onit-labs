# Теория к лабораторным ОНИТ (ЛР1–4) для Android-разработчика

Этот текст объясняет, **что у вас в проекте сделано и зачем**, простым языком. Знания Docker и бэкенда не предполагаются: там, где можно, есть сравнение с тем, что вы уже знаете из Android.

---

## 1. Зачем всё это (картина целиком)

| Лаба | Суть одной фразой |
|------|-------------------|
| **ЛР1** | Есть программа с экраном (у нас — веб-страница) и база данных; данные читаются и пишутся не «ручным SQL везде», а через **ORM**. |
| **ЛР2** | Та же программа и база упакованы так, чтобы их можно было **одной командой поднять на любом компьютере** с Docker — без ручной установки PostgreSQL и Python. |
| **ЛР3** | При каждом пуше в GitHub **автоматически** гоняются тесты и **собирается образ** приложения — как «сборка на сервере», только для сервера и контейнеров. |

## 2. ЛР1: приложение, база, ORM, CRUD

### 2.1. Что такое база данных в этом проекте

**PostgreSQL** — это отдельная программа-сервер, которая хранит таблицы и строки. Ваше приложение к ней подключается по **строке подключения** (`DATABASE_URL`: логин, пароль, хост, порт, имя базы). В коде пароль **не зашивают** — его берут из переменных окружения или из `.env` (файл с настройками только на вашей машине; в Git его не кладут).

Аналогия с Android: это как **Room** или любой другой способ хранить данные не в памяти процесса, а во внешнем хранилище — только «хранилище» живёт отдельным процессом на сервере и говорит по сети.

### 2.2. ORM (Object–Relational Mapping)

**ORM** — слой между вашими классами («сущности») и таблицами в БД. Вы описываете класс (например, «Заметка» с полями заголовок и текст), а ORM сам строит SQL для вставки, выборки, обновления и удаления.

У нас это **SQLAlchemy**: в файле модели описана таблица `notes`, а в коде вы оперируете объектами `Note`.

Зачем это на зачёте: преподаватель хочет увидеть, что вы **не пишете везде сырой SQL строками**, а используете нормальный слой доступа к данным.

### 2.3. CRUD

Это четыре базовые операции с данными:

- **C**reate — создать запись  
- **R**ead — прочитать (список или одну запись)  
- **U**pdate — изменить  
- **D**elete — удалить  

В проекте это видно на сайте: список заметок, форма «новая», «изменить», кнопка «удалить».

### 2.4. Почему интерфейс — веб, а не Android-приложение

По заданию нужен **GUI** с минимальным оформлением. Веб (HTML + формы) удобен тем, что **один и тот же код** потом легко крутится в Docker на сервере без экрана телефона. Смысл тот же, что у экрана со списком и формой: пользователь видит данные и нажимает кнопки.

Технически: **FastAPI** принимает HTTP-запросы, **Jinja2** подставляет данные в HTML-шаблоны. Точка входа процесса — **uvicorn** (HTTP-сервер для Python), как «поднятый сервер» для вашего приложения.

## 3. ЛР2: Docker и Docker Compose

### 3.1. Зачем Docker, если есть Android Studio и эмулятор

На Android вы привыкли: есть **проект**, **сборка** (Gradle), **артефакт** (APK/AAB), **среда выполнения** (устройство или эмулятор с определённой версией Android).

В мире серверов и бэкенда Docker решает похожую задачу: **«упаковать приложение вместе с тем, как его запускать»**, чтобы на другой машине не ставить вручную Python 3.12, PostgreSQL 16 и десяток библиотек.

### 3.2. Образ и контейнер (самое важное)

- **Docker-образ** — это как **неизменяемый слепок** файловой системы + метаданные «какую команду запустить». Аналогия: не APK, а скорее **образ системного раздела** + инструкция «что выполнить при старте» — но легче и быстрее, чем полная виртуальная машина.
- **Контейнер** — это **запущенный экземпляр** образа. Один образ → можно запустить много контейнеров (как много процессов из одного APK на разных устройствах — грубо).

Контейнер **изолирован**: у него свой сетевой стек, свои процессы, но он делит **ядро** ОС с хостом (в отличие от эмулятора Android, который тяжелее).

### 3.3. Dockerfile и multi-stage

**Dockerfile** — рецепт «как собрать образ»: с какого базового образа начать, какие файлы скопировать, какие команды выполнить (`RUN pip install …`).

**Multi-stage** — в одном Dockerfile **два (или больше) этапа** `FROM`:

1. На первом этапе ставят всё тяжёлое для **сборки** (компиляторы, лишние зависимости).
2. На финальном этапе в образ попадает только **нужное для запуска** (например, уже установленные пакеты копируют из первого этапа, исходники приложения).

Зачем: **меньше размер образа**, меньше поверхность для атак, быстрее выкатывать. Это прямо требование в ЛР2.

### 3.4. Docker Compose

**Compose** — файл (у нас `docker-compose.yml`), где описано **несколько сервисов** и как они связаны:

- сервис **`db`** — контейнер PostgreSQL;
- сервис **`app`** — ваше приложение.

Вы пишете одну команду `docker compose up`, и Docker **поднимает сеть**, тома, порты и оба контейнера в нужном порядке.

### 3.5. Почему приложение и БД в разных контейнерах

Так по заданию: **один контейнер — одна роль**. База живёт своим жизненным циклом (данные в **volume**, чтобы не пропали при пересоздании контейнера приложения). Приложение можно пересобрать и перезапустить, не трогая данные в Postgres.

### 3.6. Переменные окружения и `.env`

**Нельзя** класть пароли внутрь образа или в Git. Их передают при запуске:

- в **`.env`** лежат `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB` (Compose подставляет их в Postgres и в строку подключения приложения);
- в `docker-compose.yml` для приложения строка `DATABASE_URL` **собирается** из этих переменных и хоста **`db`** (имя сервиса в сети Compose — это и есть «адрес» базы **изнутри** контейнера приложения).

Важно: **`127.0.0.1` внутри контейнера приложения — это сам контейнер**, а не ваш компьютер. Поэтому хост базы для приложения в Docker — **`db`**, а с **вашего Windows** к Postgres попадают через проброс порта **`5433:5432`**.

### 3.7. `depends_on` и healthcheck

**`depends_on`** говорит: сначала подними зависимый сервис. Но Docker **не ждёт**, пока Postgres реально начнёт принимать соединения — только пока **контейнер** стартанул.

Поэтому добавляют **healthcheck**:

- у **БД** проверка вида «`pg_isready`» — Postgres готов принимать клиентов;
- у **приложения** — запрос на `http://127.0.0.1:8000/health` **изнутри** контейнера приложения (для самого приложения это его локальный порт).

В Compose у приложения стоит **`depends_on` с условием `service_healthy`** для `db`: приложение стартует только когда БД прошла проверку.

На защите можно сказать: «`depends_on` задаёт порядок, healthcheck гарантирует готовность сервиса, а не только факт запуска контейнера».

## 4. ЛР3: CI/CD и GitHub Actions

### 4.1. Что такое CI и CD словами

- **CI (Continuous Integration)** — при каждом изменении в репозитории **автоматически** ставят зависимости, **запускают тесты**, собирают артефакты. Если тест упал — автор сразу видит, что сломалось.
- **CD (Continuous Delivery / Deployment)** — после успешных тестов **автоматически** (или почти) доставляют результат дальше: у нас это **сборка Docker-образа и выкладка в GitHub Container Registry (GHCR)**.

Аналогия: это как **CI на Bitrise/GitHub Actions для Android**, только вместо `./gradlew assemble` — `docker build` и пуш образа.

### 4.2. Файл workflow

В репозитории лежит `.github/workflows/ci-cd.yml`. Там описано:

- **когда** запускать (push / pull request в ветки `main` или `master`);
- **job `test`**: клонировать код, поставить Python, поднять **сервис PostgreSQL** на раннере GitHub, выполнить `pytest`;
- **job `build`**: только на **push** (не на каждый PR), после успешных тестов — **собрать образ** и **запушить** в GHCR с тегами (ветка, SHA, `latest` для основной ветки).

### 4.3. Секреты и переменные

Пароли и ключи в код не пишут. В GitHub: **Settings → Secrets and variables → Actions**.

В проекте есть `docs/SECRETS.md` с именами. В workflow для тестов используются значения по умолчанию, если секреты не заданы — чтобы форк/репозиторий без настройки тоже получал зелёный пайплайн; смысл ЛР3 (вынести чувствительные вещи из кода) вы на защите формулируете так: «локально — `.env`, в CI — Secrets/Variables, в образ пароль не зашит».

## 5. ЛР4: Nginx и балансировка

### 5.1. Роль Nginx

**Nginx** — веб-сервер и **обратный прокси**. Клиент (браузер) стучится **только на Nginx** (у нас порт **8080** на хосте). Nginx по правилам **пересылает** запрос на один из бэкендов.

### 5.2. Upstream и round-robin

В конфиге `lr4/nginx.conf` блок **`upstream`** перечисляет три сервера: `app1`, `app2`, `app3` (имена сервисов в Compose — внутренние DNS-имена).

**Round-robin** — алгоритм по умолчанию: запросы идут **по кругу** на первый, второй, третий сервер и снова с первого. Поэтому при нескольких запросах на одну и ту же ссылку в HTML может появляться «Нода 1», «Нода 2», «Нода 3».

Почему в браузере иногда «не переключается»: браузер может **держать одно соединение**. Для демонстрации удобнее несколько запросов подряд (скрипт в `DEMO.md`).

### 5.3. Откуда берётся номер ноды

В каждом из трёх контейнеров приложения задана переменная **`NODE_ID`** (1, 2 или 3). Приложение читает её и вставляет в HTML. Это и есть «информация, позволяющая идентифицировать ноду».

## 6. Как поднять и остановить всё (шпаргалка)

Из корня `labs` (ЛР1–2):

```bash
docker compose up -d --build
```

Браузер: `http://127.0.0.1:8000/`

Остановка и удаление контейнеров сети (данные БД в volume по умолчанию сохраняются, если не указать `-v`):

```bash
docker compose down
```

Полный сброс включая том с данными Postgres:

```bash
docker compose down -v
```

ЛР4 из папки `lr4`:

```bash
docker compose up -d --build
```

Браузер: `http://127.0.0.1:8080/`  
Остановка: `docker compose down` из папки `lr4`.

Сейчас по вашей просьбе всё **остановлено** (`docker compose down` в обоих местах).

## 8. Мини-глоссарий

| Термин | Простыми словами |
|--------|------------------|
| **Контейнер** | Изолированный процесс с «своей» файловой системой из образа. |
| **Образ** | Слепок для создания контейнеров. |
| **Volume** | Именованный диск для данных, который переживает пересоздание контейнера. |
| **Порт** | «Дверь» на машине; `8000:8000` значит: порт 8000 на вашем ПК ведёт в порт 8000 в контейнере. |
| **ORM** | Слой между объектами в коде и таблицами в БД. |
| **CI/CD** | Автоматические тесты и сборка/выкладка при изменениях в Git. |
| **Reverse proxy** | Сервер перед приложениями: принимает внешние запросы и пересылает внутрь. |
| **Round-robin** | Распределение запросов по очереди между серверами. |

Если что-то из практики не сходится — смотрите **логи**: `docker compose logs` или `docker compose logs app` из соответствующей папки.
